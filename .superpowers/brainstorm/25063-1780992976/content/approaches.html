How should the mission system relate to the existing chat engine?

A: Chat-First Mission Orchestration

Recommended ★

Keep the existing useChat/streamText engine as the base. Build a MissionOrchestrator component that wraps the chat in a mission flow. Chat is hidden during cards/challenge mode; surfaced when student asks for help.

Pros

  • Leverages all existing infra (DB, auth, streaming)
  • LLM teaches lessons & generates challenges
  • Least code to delete
  • Mission wrapper is a new component, not a rewrite

Cons

  • Still fundamentally chat-driven underneath
  • Chat session persists even when hidden
  • Mission progress is derived from chat messages

B: Separate Mission Engine

New MissionEngine state machine. Missions have a deterministic lifecycle: INTRO → CARDS → GATE → CHALLENGE → RESULTS. Chat engine is only mounted when student asks for help. LLM called only for lesson generation and reteaching.

Pros

  • Clean separation — mission engine and chat don't share state
  • Snappiest UX — cards are instant (pre-authored)
  • No streaming latency for core flow
  • Most Duolingo-like feel

Cons

  • Two independent state systems to maintain
  • More upfront build work
  • Chat help feels disconnected from mission

C: Unified Hybrid

Single MissionProvider context wraps the entire app. Mission state and chat state live together. The mission controls what's visible; the chat fires lesson/challenge requests. Mission progress is saved directly to DB (not derived from messages).

Pros

  • Single source of truth for progress
  • Mission journey is first-class
  • Progress saved independently of chat history

Cons

  • New context provider changes the entire app tree
  • Refactor of most client components
  • Friction with existing ActiveChatProvider

Tap to select your preferred approach. My recommendation is A — it maximises reuse, is the safest path, and the mission feel comes from the UI layer, not the engine.
